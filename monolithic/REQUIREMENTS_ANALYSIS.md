# 모놀리식 구현 분석 — 비즈니스 요구사항을 어떻게 기술적으로 풀었는가

> 대상: `monolithic` 모듈 (Spring Boot / Kotlin / JPA(MySQL) + Redis)
> 작성일: 2026-06-24
> 기준: `README.md` 요구사항 1~5
> 성격: **학습용 기록.** "무엇이 부족한가"보다 "각 요구사항을 어떤 기술 선택으로 풀어냈는가"에 초점을 둔다.

---

## 1. 한눈에 보기

| # | 요구사항      | 기술적 해법                           | 핵심 키워드                         |
|---|-----------|----------------------------------|--------------------------------|
| 1 | 주문 데이터 저장 | JPA 엔티티 + Repository 영속화         | `Order` / `OrderItem`          |
| 2 | 재고관리      | 도메인 객체가 차감·검증을 캡슐화               | `Product.decreaseStock`        |
| 3 | 포인트 사용    | 도메인 객체가 차감·검증, 주문 트랜잭션에 합류       | `Point.usePoint`               |
| 4 | 데이터 정합성   | **단일 로컬 트랜잭션**으로 원자성 확보          | `@Transactional` 전파 `REQUIRED` |
| 5 | 동일 주문 1회  | **분산락(동시성) + 상태 플래그(재요청)** 2축 방어 | Redis Lock + `OrderStatus`     |

이 모듈의 학습 포인트는 **"모놀리식에서는 정합성과 멱등성을 어떤 도구로 푸는가"**, 그리고 그것이 **나중에 서비스를 분리(MSA)했을 때 왜 어려워지는가**를 체감하는 데 있다.

---

## 2. 처리 흐름

주문을 **생성(create)** 과 **결제(place)** 두 단계로 분리한 것이 설계의 출발점이다.

```
POST /order         → OrderService.createOrder      (@Transactional)
                      └ Order(status=CREATED) 저장 + OrderItem 저장 → orderId 반환

POST /order/place   → OrderFacade.placeOrder
                      ├ Redis 분산락 획득 (order:monolithic:{orderId})
                      └ OrderService.placeOrder      (@Transactional)
                          ├ order 조회 / 이미 COMPLETED 면 즉시 return  (멱등)
                          ├ 상품별 ProductService.buyProduct  → 재고 차감 + 금액 합산
                          ├ PointService.usePoint(...)         → 포인트 차감
                          └ order.complete()                   → 상태 전이
                      (finally) Redis 락 해제
```

생성과 결제를 나눈 덕분에 **"결제는 어떤 주문(orderId)에 대해 1회"** 라는 명확한 멱등성 단위가 생긴다. 요구 5의 해법이 여기서 출발한다.

## 3. 요구사항별 기술적 해법

### 요구 1 — 주문 데이터 저장

- `Order`(`orders`), `OrderItem`(`order_items`)를 JPA 엔티티로 모델링하고 `JpaRepository`로 영속화.
- 주문(헤더)과 주문 항목(라인)을 **별도 엔티티**로 나눠, 한 주문에 여러 상품을 담는 구조를 표현.
- `createOrder`에서 `save`로 주문을 먼저 저장해 `orderId`를 확보한 뒤, 그 id로 `OrderItem`들을 `saveAll`.

### 요구 2 — 재고관리

- 재고 차감 로직을 서비스가 아니라 **도메인 객체(`Product`) 안**에 둔 것이 핵심.
  ```kotlin
  fun decreaseStock(quantity: Long) {
      require(quantity > 0) { ... }
      require(stock >= quantity) { "재고가 부족합니다" }  // 비즈니스 불변식
      stock -= quantity
  }
  ```
- "재고는 음수가 될 수 없다"는 **불변식(invariant)을 객체 스스로 지키게** 하여, 어디서 호출하든 규칙이 새지 않는다.
- 서비스(`ProductService.buyProduct`)는 조회·차감·금액계산을 조율(orchestration)만 담당.

### 요구 3 — 포인트 사용

- 재고와 동일한 패턴: 잔액 검증·차감을 **도메인 객체(`Point`)** 가 캡슐화.
  ```kotlin
  fun usePoint(amount: Long) {
      require(this.amount >= amount) { "포인트가 부족합니다" }
      this.amount -= amount
  }
  ```
- 포인트 차감이 **주문 결제와 같은 트랜잭션 안**에서 일어나는 것이 요구 4와 직결된다(아래).
- 현재는 사용자를 `usePoint(1L, ...)`로 고정 — 사용자 식별/인증을 도입하기 전 **학습 단계의 의도적 단순화**다. (관심사를 "정합성·멱등성"에 모으기 위한 선택)

### 요구 4 — 데이터 정합성 (이 모듈의 핵심)

- `OrderService.placeOrder`가 `@Transactional`이고, 그 안에서 호출하는 `ProductService.buyProduct`·`PointService.usePoint`도
  `@Transactional`이다.
- Spring의 기본 트랜잭션 전파는 `REQUIRED` → **이미 진행 중인 트랜잭션이 있으면 새로 만들지 않고 합류**한다.
- 결과적으로 **재고 차감 + 포인트 차감 + 주문 상태 전이가 하나의 물리 트랜잭션**으로 묶인다.

```
[ 단일 트랜잭션 ]
   재고 차감 ──▶ 포인트 차감 ──▶ order.complete()
        └──────── 어느 하나라도 실패하면 전부 ROLLBACK ────────┘
```

- 즉, 포인트가 부족해 예외가 나면 **앞서 차감한 재고까지 자동 원복**된다. 별도의 보상(compensation) 코드가 필요 없다.
- **학습 포인트**: 지금은 단일 DB라서 "공짜로" 정합성이 보장된다. 하지만 주문·재고·포인트를 별도 서비스/DB로 분리하면 이 단일 트랜잭션이 사라지고, **Saga·분산 트랜잭션** 같은 기법이
  필요해진다. 이 레포의 이름(`distributed-transaction`)이 가리키는 지점이 바로 여기다.

### 요구 5 — 동일 주문 1회 보장 (멱등성)

중복 요청은 성격이 둘이라, 방어도 **2축**으로 설계했다.

**(1) 동시(병렬) 중복 — Redis 분산락**

- 같은 `orderId`에 대해 `order:monolithic:{orderId}` 키로 락을 잡는다(`setIfAbsent`).
- 동시에 10개 요청이 와도 락을 잡은 1개만 진입하고, 나머지는 락 실패로 거절된다. (`OrderFacadeConcurrencyTest`가 검증)

**(2) 순차(시간차) 중복 — 상태 플래그**

- 이미 처리된 주문이면 `order.isCompleted()`에서 즉시 `return`. 락이 풀린 뒤 다시 와도 재처리되지 않는다.

**설계의 핵심 — 락을 트랜잭션 "바깥"에 둔 이유**

- 락 획득/해제는 `OrderFacade`(트랜잭션 없음)에서, 실제 처리는 `@Transactional`인 `OrderService`에서 한다.

```
락 획득  →  [ 트랜잭션 시작 ... 커밋 ]  →  락 해제
```

- 만약 락을 트랜잭션 안에서 잡고 풀면, **"락 해제 → 아직 DB 커밋 전"** 인 찰나에 다음 요청이 들어와 `CREATED` 상태를 읽고 또 처리할 수 있다.
- 락을 트랜잭션 밖으로 빼면 **커밋 완료 후에야 락이 풀리므로**, 다음 요청은 반드시 커밋된 `COMPLETED`를 본다. → 두 축의 방어가 빈틈없이 맞물린다.

## 4. 이 구현에서 배울 핵심 설계 원칙

1. **불변식은 도메인 객체 안에서** — 재고/포인트 검증을 엔티티가 직접 지켜, 호출 위치와 무관하게 규칙이 보장된다.
2. **정합성은 트랜잭션 경계로** — 여러 도메인 변경을 하나의 `@Transactional` 흐름(전파 `REQUIRED`)으로 묶어 원자성을 얻는다.
3. **멱등성은 "동시성"과 "재요청"을 분리해서** — 분산락(동시)과 상태 플래그(재요청)는 막는 대상이 다르며, 둘 다 필요하다.
4. **락과 트랜잭션의 경계 순서가 정확성을 좌우** — "커밋 후 락 해제"를 지키려 락을 트랜잭션 바깥에 배치했다.

## 5. 현재 단계의 단순화와 다음 학습 주제

학습 흐름상 "지금은 이렇게 단순화했고, 다음 단계에서 이렇게 발전시킨다"의 관점으로 정리한다. (세부 진행은 `IMPROVEMENTS.md`)

| 주제       | 현재 단계의 단순화              | 다음 단계에서 다룰 것                     |
|----------|-------------------------|----------------------------------|
| 분산락 견고성  | TTL 없는 단순 `setIfAbsent` | 만료시간 부여 → 프로세스 다운 시 락 자동 회수      |
| 사용자 식별   | `userId` 고정(1L)         | 주문 주체를 `Order`에 담아 실제 사용자 포인트 차감 |
| 락 해제 안전성 | 값 비교 없이 `delete`        | 소유권 검증(Lua) — 내 락만 해제            |
| 정합성 범위   | 단일 DB·단일 트랜잭션           | 서비스 분리 후 Saga/분산 트랜잭션으로 확장       |

---

## 6. 결론

이 모놀리식 모듈은 5개 요구사항을 **(1) 도메인 캡슐화로 비즈니스 규칙을, (2) 단일 트랜잭션으로 정합성을, (3) 분산락+상태 플래그로 멱등성을** 푸는 방식으로 구현했다. 특히 **정합성을 "공짜로" 얻는
모놀리식의 구조**와 **락·트랜잭션 경계 설계**는, 이후 서비스를 분리했을 때 무엇이 어려워지는지를 비교하기 위한 좋은 기준점이 된다.
